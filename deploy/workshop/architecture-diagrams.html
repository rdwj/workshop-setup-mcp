MCP Ecosystem on OpenShift — Architecture

Two views of what the workshop builds: how a developer's token travels and transforms, and the end-state system. Companion to the workshop guide.

1. Per-user auth & token flow

The blue token never changes. The JWT a developer gets from Keycloak is the same credential the Kubernetes API eventually validates — External OIDC puts them in one trust domain. The amber path is the one transformation in the system: the per-route policy trades the JWT at Vault for that user's GitHub PAT. Any backend route without its own policy gets the fail-closed default: JWT required, Authorization stripped. One subtlety: the router rewrites the Host header before auth runs, so the client-plane policy attaches to the broker’s HTTPRoute (mcp-gateway-route) rather than the listener.

2. End-state system architecture

The trust triangle is the core of the design. Keycloak anchors three trust relationships (dashed lines): the gateway trusts its JWKS (Module 8), the Kubernetes API trusts it via External OIDC (Module 7), and Vault trusts it via JWT auth (Module 11). One realm, one login, three enforcement points. Authorization happens at four layers: wristband + VirtualMCPServer filter what a user sees; per-route Rego enforces what they can call; K8s RBAC / GitHub scopes bound what the call can do; the audit log attributes it.

Removing a Keycloak tool role degrades all four authorization layers on the user's next token — no gateway or policy changes required. Full background: docs/per-user-mcp-identity-fix.md in the repo.
