# Add Memory

Wire MemoryHub integration into the agent for persistent memory across runs. This command configures both the SDK path (`self.memory` for agent code) and the MCP path (MemoryHub tools for the LLM).

**Prerequisite: `/create-agent` must have been run first.** Verify `src/agent.py` exists and contains a BaseAgent subclass before proceeding.

**This is optional.** Not every agent needs persistent memory. If the agent handles stateless, one-shot tasks, skip this. Memory is valuable when:
- The agent needs to remember things between separate runs or conversations
- The agent shares context with other agents via scoped memories
- The agent should learn from past interactions (preferences, corrections, patterns)

## Process

### Step 1: Assess Memory Needs

Ask the developer:

1. **What should the agent remember?** Get specific examples: user preferences, past decisions, discovered facts, error patterns, domain knowledge.
2. **What scope level?** MemoryHub supports:
   - `user` — personal to the user interacting with the agent
   - `project` — shared across agents working on the same project
   - `role` — shared across agents with the same role
   - `organizational` — shared across a team or organization
   - `enterprise` — mandated policies visible to all agents
3. **Should the agent read, write, or both?** Some agents only consume memories written by others. Some only produce memories for downstream agents. Most do both.
4. **Is a MemoryHub server available?** The agent needs a running MemoryHub instance to connect to. If one is not deployed yet, note it — the integration will degrade gracefully to a no-op client until the server is available.

### Step 2: Run memoryhub config init

If the `memoryhub` CLI is available, run:

```bash
memoryhub config init
```

This generates two files:
- `.memoryhub.yaml` — behavioral configuration (committed to source)
- `.claude/rules/memoryhub-loading.md` — rules for memory hygiene (committed to source)

If `memoryhub` is not installed, create `.memoryhub.yaml` manually:

```yaml
# MemoryHub integration configuration
# Generated by: memoryhub config init
#
# This file is committed to source control. It defines how the agent
# interacts with MemoryHub. The actual API key and server URL are
# injected via environment variables at deploy time.

loading: lazy

server_url: ${MEMORYHUB_URL:-http://memoryhub:8080}

scopes:
  - user
  - project

memory_hygiene:
  write_preferences: true
  write_decisions: true
  write_architectural_choices: true
  skip_ephemeral: true
```

Adjust the `scopes` list based on what the developer said in Step 1.

### Step 2b: Update the Containerfile

Add `.memoryhub.yaml` to the container image so it is baked in at build time.

In `Containerfile`, find the line `COPY AGENTS.md ./` and add the following line immediately after it:

```dockerfile
COPY .memoryhub.yaml   ./
```

Also extend the `chmod` RUN command to include `.memoryhub.yaml`. Replace the entire multi-line `RUN chmod ...` block with:

```dockerfile
RUN chmod -R g=u,o=r src/ tools/ prompts/ skills/ rules/ agent.yaml .memoryhub.yaml \
    && find src/ tools/ prompts/ skills/ rules/ -type d -exec chmod g=u,o=rx {} +
```

**Important**: Keep both lines of the `RUN` command. The `find` line sets directory execute bits, which are required for directory traversal in the container.

Remove (or leave in place — it is harmless) the comment that `/add-memory` would add this line.

### Step 3: Update agent.yaml

The `memory` section in `agent.yaml` should already point to `.memoryhub.yaml`:

```yaml
memory:
  config_path: .memoryhub.yaml
```

If it does not exist, add it. This is the only configuration BaseAgent needs — the memory module reads `.memoryhub.yaml` at startup and handles everything else.

If the developer wants the MemoryHub MCP tools available to the LLM (so the LLM can read/write memories during tool calling), add the MemoryHub MCP server to the `mcp_servers` list:

```yaml
mcp_servers:
  - url: ${MEMORYHUB_MCP_URL:-http://memoryhub:8080/mcp}
```

This makes MemoryHub's 15 tools available as LLM-callable tools (plane 2). The SDK path (`self.memory`) works independently of this — it uses the MemoryHub Python SDK directly.

### Step 4: Install the memoryhub Package

Add `memoryhub` to the project dependencies if not already present:

1. Check `pyproject.toml` — it may already be in an optional `[project.optional-dependencies.memory]` group.
2. If missing, add it to the appropriate dependency group.
3. Install: `pip install -e ".[memory]"` or `pip install memoryhub`.

The memory module (`src/fipsagents/baseagent/memory.py`) lazy-imports `memoryhub`, so the agent runs fine without it installed — it just falls back to `NullMemoryClient`. But for the integration to actually work, the package must be present.

### Step 5: Show Usage Patterns

Demonstrate how to use memory in `src/agent.py`:

**Reading memories in step():**

```python
async def step(self) -> StepResult:
    # Search for relevant memories at the start of a task
    query = self.messages[-1]["content"] if self.messages else ""
    memories = await self.memory.search(query)

    # Include relevant memories in the system prompt or context
    if memories:
        memory_context = "\n".join(
            f"- {m.get('content', '')}" for m in memories
        )
        self.messages.insert(0, {
            "role": "system",
            "content": f"Relevant context from previous sessions:\n{memory_context}",
        })
    ...
```

**Writing memories after completing a task:**

```python
    # After producing a result, save key learnings
    await self.memory.write(
        content=f"User preference: {preference}",
        scope="user",
        weight=0.8,
    )
```

**Updating an existing memory:**

```python
    # When a preference or fact changes, update instead of creating a duplicate
    await self.memory.update(
        memory_id=existing_memory_id,
        content=f"Updated preference: {new_preference}",
    )
```

These are patterns, not prescriptions. The developer decides which memories are worth persisting based on their agent's purpose.

### Step 6: Handle the API Key

The MemoryHub API key should never be hardcoded. The standard location is `~/.config/memoryhub/api-key` for local development. In production (OpenShift), it comes from a Secret mounted as an environment variable.

For local development, verify the key file exists:
```bash
cat ~/.config/memoryhub/api-key 2>/dev/null && echo "(key found)" || echo "No API key found at ~/.config/memoryhub/api-key"
```

For production deployment, add to the Helm chart's Secret template or document that the `MEMORYHUB_API_KEY` environment variable must be set.

### Step 7: Verify the Integration

Run a quick check to see if the memory client initializes:

```bash
python -c "
import asyncio
from fipsagents.baseagent.memory import create_memory_client

async def check():
    client = await create_memory_client('.memoryhub.yaml')
    print(f'Client type: {type(client).__name__}')
    # If MemoryClient, the server was reachable
    # If NullMemoryClient, either config is missing, server is down, or package not installed

asyncio.run(check())
"
```

Expected outcomes:
- `MemoryClient` — fully connected, integration working
- `NullMemoryClient` — graceful degradation; check the logs for the reason (missing config, missing package, server unreachable)

Both are valid states. The agent works either way.

### Step 8: Run Tests

Run `make test` to verify the integration does not break anything. The test suite's memory tests use `NullMemoryClient` by default, so they pass regardless of whether a real MemoryHub server is available.

### Step 9: Summary

Tell the developer:
- Whether `.memoryhub.yaml` was created or already existed
- Configured scopes
- Whether the MCP path (LLM tools) was enabled
- Whether `self.memory` (SDK path) is available
- Current client status (MemoryClient vs NullMemoryClient and why)
- How to set up the API key for local development
- How to configure the key for production (OpenShift Secret)
